# 开发计划：自适应学习路径产品（美国5年级 MVP）

## 1. 产品定位

**目标**：基于 Marble Skill Taxonomy 数据集，构建一个能够诊断学生知识掌握情况、动态规划学习路径的自适应学习系统。

**MVP 范围**：
- 目标用户：美国 5 年级学生（约 10-11 岁）
- 学科范围：先聚焦**数学**（5 年级年龄段内共 211 个知识点），打通"诊断 → 路径推荐 → 练习 → 重新评估"全链路后再横向扩展到其他学科
- 之所以选数学优先：知识点间的先修关系明确、答案客观可判分，不依赖主观评分，适合快速验证算法

**数据基础**（已核实）：
- 与 5 年级年龄段（10-11 岁）有交叠的知识点共 678 个，其中数学 211、科学 211、英语 166
- 这些知识点关联的先修依赖共 1,519 条，其中约 **613 条（40%）指向年级以下的知识点** —— 说明相当比例的学生需要"回溯补基础"，这正是自适应系统要解决的核心问题

## 2. 关键设计决策（已确认）

| 决策点 | 选择 | 说明 |
|---|---|---|
| 题库来源 | LLM 辅助生成 | 用 Claude API 基于 topic 的 `description`/`evidence`/`assessmentPrompt` 批量生成结构化题目，人工抽检质量，速度快，适合验证 MVP |
| 技术栈 | Web App | Next.js（React + Tailwind）前端 + Node.js 后端 + Postgres，先在浏览器验证产品假设，后续可扩展移动端 |
| 掌握度算法 | 简单规则起步 | 连对/连错触发状态变化，不做贝叶斯知识追踪（BKT），表结构预留升级空间 |

## 3. 技术架构

### 3.1 数据层
- 将 `data/topics.json`、`data/dependencies.json`、`data/curriculum-standards.json`（可选）导入 Postgres
- 核心表：`topics`、`dependencies`、`topic_standards`
- 图查询（找先修链、找可解锁节点）用递归 CTE 实现，无需引入专用图数据库

### 3.2 题库层
- 新增 `questions` 表：`topic_id`、题干、选项、正确答案、干扰项说明
- 生成流程：Claude API 按 topic 批量生成结构化题目 → 存入表 → 人工抽检 10-20% 质量 → 上线
- 覆盖范围：5 年级数学 211 个 topic + 其向下延伸的全部 prerequisite topic（用于诊断回溯）

### 3.3 学习者模型（简单规则版）
表：`student_topic_mastery(student_id, topic_id, status, correct_streak, wrong_streak, last_seen)`

规则：
- 连续答对 3 题 → 标记 `mastered`
- 连续答错 2 题 → 标记 `gap`，将其 `hard` prerequisite 加入路径队列优先学习
- `soft` prerequisite 仅在多个 topic 同时卡住时才触发插入
- 该模型结构简单，后续可无痛升级为贝叶斯知识追踪（BKT），表结构基本不用改，只替换算分逻辑

### 3.4 诊断引擎
- 学生首次进入 → 分层诊断测试：先测年级水平 topic，答错则回溯测其 prerequisite，逐步定位"知识前沿"（mastery frontier）
- 不做 1590 题全测，用图结构做自适应抽样，减少诊断题量

### 3.5 路径规划算法
- 输入：学生当前掌握状态图
- 排序信号：
  1. 图结构约束——所有 `hard` prerequisite 已掌握才可解锁
  2. `centrality` 字段（数据集已提供）——优先推荐图中枢纽知识点，学会后能解锁更多后续内容
  3. 年级/年龄贴近度——优先本年级内容，年级以下缺口标注为"补基础"而非"降级"
- 输出：一条可解释的学习路径（不做黑盒推荐，家长/教师能看懂"为什么先学这个"）

### 3.6 自适应练习循环
- 学生做题 → 更新 mastery → 触发路径重算
- 连续错误 → 自动插入 prerequisite 补习节点
- 连续正确 → 跳过已掌握内容，加速前进

### 3.7 前端
- 学生视图：单一"今天学什么"卡片流 + 简易进度图（可用 react-flow 类库做知识图可视化，参考数据集 README 中的 3D 图谱，MVP 阶段用简化 2D 即可）
- 家长/教师视图：结合 `clusters.json` 的通俗摘要生成进度报告

## 4. 时间线（约 10-12 周到可用 MVP）

| 阶段 | 内容 | 时间 |
|---|---|---|
| 1. 数据基础设施 | 导入数据到 Postgres，建 schema，写 5 年级数学子图提取脚本 | 1 周 |
| 2. LLM 题库生成 | 为 211 个 5 年级数学 topic + 其向下延伸的 prerequisite topic 批量生成题目，人工抽检 | 2 周 |
| 3. 诊断引擎 | 分层诊断测试逻辑（先测年级水平，错则回溯） | 1.5 周 |
| 4. 路径算法 | 掌握图 + centrality 排序 + 可解锁 topic 计算 | 1.5 周 |
| 5. 练习循环 API | 做题 → 更新 mastery → 重算路径 的后端接口 | 1.5 周 |
| 6. 前端 MVP | 学生做题界面 + "今天学什么"卡片流 + 简易进度图（可与阶段 4/5 并行） | 3 周 |
| 7. 内部测试 + 打磨 | 找几个 5 年级学生真实测试，调整题目难度和路径逻辑 | 1 周 |

## 5. 第一个可交付里程碑（2-3 周内，垂直切片）

在投入大规模题库生成和前端开发之前，先跑通一条子链验证架构假设：

1. 选数学里一条子链（如"分数"相关的 15-20 个 topic）
2. 导入这部分数据到 Postgres
3. LLM 出题（覆盖这 15-20 个 topic 及其 prerequisite）
4. 命令行/最简页面跑通：诊断 → 做一道题 → 更新掌握状态 → 推荐下一题
5. 验证通过后再扩大到全部 211 个 5 年级数学 topic

## 6. 开放问题 / 后续需要澄清

- 年级下探范围：诊断时最多回溯到几个年级去找知识缺口？（建议先设上限，如最多回溯 3 个年级，避免诊断过程过长）
- 题目形式：MVP 阶段是否只做选择题，还是也需要填空/简答（简答会引入判分复杂度）
- 数据许可：`curriculum-standards.json` 中标准原文的许可证各不相同（见项目 `PROVENANCE.md`），若产品对外发布需确认展示范围是否合规
- 学科扩展顺序：数学验证通过后，下一个学科选英语（166 个 topic）还是科学（211 个 topic）

## 7. 参考数据（来自 Marble Skill Taxonomy 数据集）

- 总计 1,590 个知识点，8 大学科，3,221 条先修依赖
- 数据文件：`data/topics.json`、`data/dependencies.json`、`data/curriculum-standards.json`、`data/clusters.json`
- 校验脚本：`node scripts/validate.mjs`
